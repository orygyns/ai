10 — BYOK System (Bring Your Own Key)
ORYGYNS never handles billing. Users paste their own Groq API key (gsk_...), which is encrypted and stored. All AI calls are proxied through the user's own key server-side. ORYGYNS displays usage; users pay Groq directly.
User visits orygyns.ai
→
Paste gsk_... key
→
Test call to Groq /models
→
Key encrypted (AES-256-GCM)
→
Stored in PostgreSQL
→
All AI calls proxy via key
→
User billed by Groq directly
Key Security Rules
  • Raw key NEVER stored — encrypted immediately on receipt
  • AES-256-GCM encryption (at rest in PostgreSQL)
  • HTTPS everywhere — TLS in transit
  • Audit logs for all key access and modifications
  • Key never appears in server logs
  • Profile displays: gsk_********************
  • [Unhide] reveals full key for 30 seconds, then re-masks
User Key Management UI
  • Validate key: instant test call on paste
  • Status badge: Connected or Invalid
  • Last updated timestamp shown
  • Update key: replace with new key anytime
  • Delete key: removes from system entirely
  • Usage stats: linked to Groq Console (external)
  • CTA: [Open Groq Console →] — for direct top-up

12 — Security Requirements
Transport & Auth
  • HTTPS everywhere — forced redirect (no HTTP)
  • AES-256-GCM for Groq key encryption at rest
  • JWT auth — short-lived access + refresh tokens
  • Rate limiting per user per endpoint (Redis)
  • XSS protection via Content Security Policy headers
Data & Session
  • CSRF protection (SameSite cookies)
  • Audit logs for all key access and modifications
  • Encrypted storage — at rest and in transit
  • Secure session management
  • Auto-logout with configurable timeout
  • Raw Groq key never appears in logs or responses
